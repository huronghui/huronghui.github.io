Android View 事件分发体系

  1. 基本概念
    1. Touch 事件主要是封装在MotionEvent中（MotionEvent 似乎用到了享元模式，避免创建太多的MotionEvent 类的对象，从而占用内存）
      后续讲解 MotionEvent 类
    2. Touch 事件主要有三种 （单指滑动）
      1. ACTION_DOWN ：按下事件
      2. ACTION_UP ： 抬起事件
      3. ACTION_MOVE ：移动事件
  2. 事件总是从 ACTION_DOWN 开始的，经过一系列的ACTION_MOVE, 结束于ACTION_UP, 即从用户点击屏幕产生 DOWN 事件，移动一些距离，然后抬起，这段过程为一个完整的事件。

  1. 事件函数

    1. dispatchTouchEvent(MotionEvent ev)
      无论是View 还是ViewGroup ，事件分发的开始端都是dispatchTouchEvent 函数，从此函数开始，view 或者 viewgroup 开始了事件分发体系

    2. onInterceptTouchEvent(MotionEvent ev)
      该函数是ViewGroup的特有方法，因为ViewGroup是View的集合但同时 自己也是一个View，因此，ViewGroup 涉及到事件是否需要自己处理，如果该函数返回 true 表示拦截事件，由自己处理，如果返回 false 表示不拦截

    3. onTouchEvent(MotionEvent ev)
      处理Touch事件的核心方法，返回true 表示消费了事件，返回false表示没有消费该事件。

  2. 源代码

View 的事件分发机制相对于ViewGroup比较简单，因为View是一个单独的元素，无需向下传递事件，所以只能自己处理事件，下面为View的事件分发的源代码：

public boolean dispatchTouchEvent(MotionEvent event) {

    if (mInputEventConsistencyVerifier != null) {
        mInputEventConsistencyVerifier.onTouchEvent(event, 0);
    }

       //判断产生该 event 的Window 是否被其他窗口遮挡，如果遮挡需要过滤触摸事件
    if (onFilterTouchEventForSecurity(event)) {
        ListenerInfo li = mListenerInfo;

        //首先判断View 有没有设置OnTouchlistener，
        if (li != null && li.mOnTouchListener != null &&  (mViewFlags & ENABLED_MASK) == ENABLED && li.mOnTouchListener.onTouch(this, event)) {
            return true;
        }
        //onTouch() 优先于 onTouchEvent()函数的执行
        if (onTouchEvent(event)) {
            return true;
        }
    }

    if (mInputEventConsistencyVerifier != null) {
        mInputEventConsistencyVerifier.onUnhandledEvent(event, 0);
    }
    return false;
}

从上面的代码可以看出：

  • OnTouchEvent() 不一定会调用，如果设置了OnTouchListener，View是enable，并且OnTouchListener 的 OnTouch()返回true， 则不会调用onTouchEvent().

  • 如果View 是DISABLE, 则onTouch()不会调用

  • 如果OnTouchListener 的OnTouch()或 View 的 onTouchEvent()返回true，则dispatchTouchEvent() 返回 true；否则返回 false。

这时可能会有人要问了 onClick 方法没有执行啊？ 答案在 onTouchEvent()方法中，接着分析源码：

public boolean onTouchEvent(MotionEvent event) {
    final int viewFlags = mViewFlags;

    if ((viewFlags & ENABLED_MASK) == DISABLED) {
        if (event.getAction() == MotionEvent.ACTION_UP && (mPrivateFlags & PFLAG_PRESSED) != 0) {
            setPressed(false);
        }

        return (((viewFlags & CLICKABLE) == CLICKABLE ||
                (viewFlags & LONG_CLICKABLE) == LONG_CLICKABLE));
    }

    if (mTouchDelegate != null) {
        if (mTouchDelegate.onTouchEvent(event)) {
            return true;
        }
    }

    if (((viewFlags & CLICKABLE) == CLICKABLE ||
            (viewFlags & LONG_CLICKABLE) == LONG_CLICKABLE)) {
        switch (event.getAction()) {
            case MotionEvent.ACTION_UP:
        boolean prepressed = (mPrivateFlags & PFLAG_PREPRESSED) != 0;
                if ((mPrivateFlags & PFLAG_PRESSED) != 0 || prepressed) {
                    boolean focusTaken = false;
                    if (isFocusable() && isFocusableInTouchMode() && !isFocused()) {
                        focusTaken = requestFocus();
                    }

                    if (prepressed) {
                        setPressed(true);
                   }

                    if (!mHasPerformedLongPress) {
                        removeLongPressCallback();

                        if (!focusTaken) {

                            if (mPerformClick == null) {
                                mPerformClick = new PerformClick();
                            }
                            if (!post(mPerformClick)) {
                                performClick();
                            }
                        }
                    }

                    if (mUnsetPressedState == null) {
                        mUnsetPressedState = new UnsetPressedState();
                    }

                    if (prepressed) {
                        postDelayed(mUnsetPressedState,
                                ViewConfiguration.getPressedStateDuration());
                    } else if (!post(mUnsetPressedState)) {
                        // If the post failed, unpress right now
                        mUnsetPressedState.run();
                    }
                    removeTapCallback();
                }
                break;

            case MotionEvent.ACTION_DOWN:
                mHasPerformedLongPress = false;

                if (performButtonActionOnTouchDown(event)) {
                    break;
                }


                boolean isInScrollingContainer = isInScrollingContainer();

                if (isInScrollingContainer) {  
                    mPrivateFlags |= PFLAG_PREPRESSED;
                    if (mPendingCheckForTap == null) {
                        mPendingCheckForTap = new CheckForTap();
                    }
                    postDelayed(mPendingCheckForTap, ViewConfiguration.getTapTimeout());
                } else {
                    // Not inside a scrolling container, so show the feedback right away
                    setPressed(true);
                    checkForLongClick(0);
                }
                break;

            case MotionEvent.ACTION_CANCEL:
                setPressed(false);
                removeTapCallback();
                removeLongPressCallback();
                break;

            case MotionEvent.ACTION_MOVE:
                final int x = (int) event.getX();
                final int y = (int) event.getY();


                if (!pointInView(x, y, mTouchSlop)) {
                    // Outside button
                    removeTapCallback();
                    if ((mPrivateFlags & PFLAG_PRESSED) != 0) {
                        // Remove any future long press/tap checks
                        removeLongPressCallback();

                        setPressed(false);
                    }
                }
                break;
        }
        return true;
    }

    return false;
}

我们开始分析onTouchEvent() 的代码，这段代码比dispatchTouchEvent() 函数要长的多：

  • 从第一个 if 判断语句来看： 当View是不可用状态下时，依然是会消耗点击事件的，只是不可用状态下的点击看起来没什么用，返回的是当前的View 的 CLICKABLE 和 LONG_CLICKABLE 的设置状态。
  • 第二个 if 判断语句：如果当前的View设有代理，那么就会执行代理的onTouchEvent() 事件
  • 第三个 if 语句是最关键的，这是当前 View 对具体事件的处理所在， 从判断语句来看，只要当前的View 的 CLICKABLE 和 LONG_CLICKABLE 有一个为true，那么就会消耗这个事件，即 OnTouchEvent 事件就会返回true。
  • 此外我们可以看到，具体的事件处理函数 performClick()函数 是在 UP 的事件中执行的，而不是在 DOWN 事件中，这和我们的测试相符合。

再来看performClick的代码：

public boolean performClick() {
    sendAccessibilityEvent(AccessibilityEvent.TYPE_VIEW_CLICKED);

    ListenerInfo li = mListenerInfo;
    if (li != null && li.mOnClickListener != null) {
        playSoundEffect(SoundEffectConstants.CLICK);
        li.mOnClickListener.onClick(this);
        return true;
    }
    return false;
}

这段代码很简短，判断view 是否有设置OnClickListener ，有的话就会调用其的onClick 事件

ok，至此 View 的事件分发就解析完了，我们的结论：

  1. View 是没有onTnterceptTouchEvent 方法的，一旦有事件传递给他，那么它就会处理这个事件

  2. View 的 OnTouch 和 On TouchEvent 这两个方法都是在View的dispatchTouchEvent中调用的，onTouch优先于onTouchEvent执行。如果在onTouch方法中通过返回true将事件消费掉，onTouchEvent将不会再执行。onTouch能够得到执行需要两个前提条件，第一mOnTouchListener的值不能为空，第二当前点击的控件必须是enable的。

  3. View 的 onTouchEvent 默认是会消耗事件的(返回true)，除非他是不可点击的(即clickable 和 longclickable 同时为false)。View 的longclickable属性默认是false，但 clickable 属性是分情况的，button 的clickable 默认是true，TextView 的clickable属性默认是false，所以TextView 默认是不能消耗事件的。但我们可以通过调用 setOnckickListener() 方法将clickable 事件赋值为 true；

  4. view 的enable 属性不会影响onTouchEvent 的默认返回值，哪怕一个View 是disable状态的，只要他的clickable 和 longclickable有一个为true，那么它的onTouchEvent就返回true。

  5. onClick 事件发生的前提是 它收到了 UP 和 DOWN 事件，并且该View 是可以点击的。
