其中 activity_main 正是我们写的布局文件，我们通过 setContentView() 函数将布局加载到我们的activity中，那么看起来这个函数应该就包含了 activity 加载视图的过程了，同样我们剖析一下源码，从源码的角度来看activity 是怎样加载视图的。

来，我们一步一步分析，在activity中一共有三个setContentView，但这三个函数里面的内容大同小异，都是首先系统调用的是getWindow()函数，那问题来了，这个函数是干嘛的尼？同样，我们找到这个函数的具体实现：

就短短的一句话，返回一个 mWindow 对象，这个mWindow 对象是一个Window 类型的数据，而Window 类又是我们管理activity 视图的管理类，因此我们可以理解，getWindow()其实是返回一个视图管理类。

我们看到在PhoneWindow 中setContentView()函数的具体实现，同样有三个，第二个函数调用的还是第三个函数，所以其实他的具体实现其实只有两个，我们再看这个实现函数，发现这两个函数基本相似，不同点在于：

  1. 第一个函数调用的布局的id，我们需要根据这个布局的id 来加载视图
  2. 第二函数直接调用的View，不需要根据Id来映射到这个布局，并且直接add 就可以了。

我们在一步一步分析这段代码：首先是installDecor() 函数

这段代码很长，但大部分都是初始化DecorView 属性的，比如title，actionbar 等等。核心代码是第3行的代码，通过generateDecor()函数，我们获得了一个DecorView 的实例，同样进入该函数一探究竟：

代码很简单，只有一段话，实例化一个DecorView 对象，并且这个对象是PhoneWindow 的一个内部类，继承了FrameLayout，说明这个DecorView 是一个ViewGroup。

紧接着我们会进入另外一个函数：generateLayout(mDecor) ，该函数是干嘛用的呢？在这之前我们可以先看一下这个函数的返回值 mContentParent ,是不是很有印象，对的，正是我们在setContentView 函数中的第一个 if 判断的对象。

他的赋值是在installDecor()中实现的，因此当我们的setContentView函数第一次执行时，其值为 null ，执行过 installDecor()后，该值得到赋值，不为null，将不会在执行 installDecor() ，而是执行删除其所有的子元素操作。并且分析setContentView 下面的添加View 操作，发现我们要添加的View正是添加在mContentParent这个父View下，因此我们是不是可以大胆的猜测一下，我们的这个 mContentParent 和我们的 DecorView 是不是有什么具体的联系，并且这个联系是通过 generateLayout() 函数实现的，好啦，不多说了，直接上源码：

这段代码也比较长，代码一开始是获取 window 下的一些定义的属性，具体可以参考\frameworks\base\core\res\res\values\attrs.xml里面的属性，根据这里面的一些值，设置当前window 的一些属性，就是说我们可以理解为这边就是我们为Activity设置 theme 的地方，比如是否需要全屏，背景色等，然后将我们设置好的布局通过LayoutInflater转化成View，加入到DecorView 中，然后通过 28 行的findViewById 获取到 contentParent 视图。
我们总结一下：我们的DecorView 是一个 FrameLayout，然后根据 Theme 来选择系统布局，并将这个布局通过LayoutInflater转化一个View，添加到我们mDecor 中，这些布局文件中都会包含一个 ID 为 content 的FrameLayout，将其引用返回给 mContentParent。

那么我们现在有了mContentParent，接着要干嘛呢？再贴一下代码：

看到了么？我们会将我们写的布局文件通过inflate加入到 mContentParent 中，所以明白了吧，为什么我们的函数名会叫 setContentView， 因为我们写的布局正是添加到一个 ID 为 content 的ViewGroup 中。
接着还有一段函数：GetCallback()，这个是获取到一个 CallBack 对象，我们跟踪源码会发现，PhoneWindow 没有这个方法的实现，那么这个方法一定就是 Window 的方法了，果不其然，在 Window 中我们找到这个方法的实现：

set 和 get 都在 Window 类中，那么这个Callback 对象到底是谁呢？不卖关子了！这个对象指向的是我们的Activity类，在Activity 类中我会发现，它实现了Window的callback方法，那么这个callback的onContentChanged() 到底是干嘛的呢？ 答案是：它是一个回调，当我们布局完成之后，我们会回调 Activity 的onContentChanged 方法，因此，如果我们完成布局初始化之后想要做一些其他的事情的话，我们可以在这个函数中实现。

ok，整个的流程分析完毕，我们总结一下：

*1. 判断当前界面是否初始化了DecorView，如果没有的话，就新建一个DecorView 的实例，否则清除当前页面所有的布局。

  1. 通过Theme属性获取相应的系统布局，并通过infalter.inflater放入到我们的mDecor中，在布局中获取一个 ID 为content的FrameLayout。
  2. 将我们设置的布局，通过infalter.inflater压入 ID 为 Content 的FrameLayout中，并回调Activity的onContentChanged 方法。*
